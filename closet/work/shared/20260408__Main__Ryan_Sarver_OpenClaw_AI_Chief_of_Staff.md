## 記憶層：系統的基礎

### 核心觀點

> **Session memory 是一場騙局。** 任何把對話紀錄當作工作上下文的助理，都會在最關鍵的時刻掉鏈子。

### 兩層記憶架構

| 層次 | 內容 | 格式 |
|------|------|------|
| **Layer 1：每日筆記** | 當天發生的一切：會議、決定、任務 | `memory/YYYY-MM-DD.md` |
| **Layer 2：長期記憶** | 關鍵人物、專案、學到的教訓、已做的決定 | `MEMORY.md` |

### 純 Markdown 哲學

- 所有東西存在純粹的 Markdown 檔案裡
- 不用資料庫
- 可以直接打開看、直接編輯
- 整個資料夾可以用 git 備份
- **沒有抽象層，所以更信任系統，出問題時也修得更快**

## Kaizen：系統會自己變好

### 每週循環

| 日期 | 動作 |
|------|------|
| **週五** | Stella 自動掃描 OpenClaw 社群新做法，存入研究筆記 |
| **週日** | 一起 Review，Stella 總結本週研究，提出最值得試的改動 |

### 日常學習

- 如果 Ryan 一直修正同一件事，會被記錄
- 如果某個功能帶來的麻煩比好處多，也會被記錄
- 最後浮現為修正建議

### 系統重構紀律

> 第一版總是太複雜、太多雜訊，或在解決錯的問題。
> 一個你信任的小系統，永遠比一個你得繞來繞去才能用的大系統好。

## 會議準備與跟進

### 會前 60 分鐘
- 收到 WhatsApp 簡報
- 內容包括：
  - 與會者先前的會議記錄
  - 最近的信件往來
  - 未完成的行動項目
  - （募資情境）pipeline 階段、對方看過哪個版本的簡報

### 會後處理
- 透過 Granola API（或任何有 API 的會議記錄工具）處理會議內容
- 任務送到 Todoist（帶有專案分類和到期日）
- 別人承諾的事記錄在每人一個的 markdown 檔案裡

---

## 任務與優先順序管理

### 核心原則
- 完整的任務全貌保存在結構化的 markdown 裡
- 這是唯一的正確版本
- 近期重要事項同步到 Todoist

### 每天晚上任務掃描
- 哪些到期、哪些逾期、哪些擱置太久
- 某件事連續五天被往後推，標記這個模式
- 週二有重要會議但準備沒做完，提醒

## 人脈追蹤

### 方式
- 對每個人維護一份有組織的背景資料
- 關係歷史的最後接觸時間
- 未完成的承諾、對方在乎什麼、目前進度

### 價值
> 本質上就是把記憶應用在人際關係上，隨著時間推移複利效果最明顯的環節。

## 關鍵設計原則：判斷力 vs 可預測性

### 核心規則
> **LLM 處理判斷，Scripts 處理其他一切。**

| 類型 | 處理方式 |
|------|----------|
| **固定邏輯** | Python Scripts（讀檔案、呼叫 API、發訊息、比較時間戳記）|
| **需要推理** | LLM（綜合、排列優先順序、草擬文字）|

> 把固定邏輯的工作丟給 LLM，事情會以不可預測的方式壞掉，你就再也不信任這個系統了。
