## 🔧 項目 1: MCP SDK 研究

### 實作過程
- 研究 MCP v1.26.0 和 fastmcp v3.1.1 架構
- 分析核心模組：Server、FastMCP、Stdio Transport、Streamable HTTP Transport
- 建立類型映射表（MCP ↔ OpenClaw）

### 關鍵決策
- 採用 FastMCP 作為主要 API，簡化開發
- 使用低階 Server API 以獲得更多控制權

---

## 🔧 項目 2: Stdio Server 管理

### 實作過程
- 實現 `StdioServerManager` 類
- 管理子進程生命週期
- 實現 JSON-RPC 請求/響應機制
- 使用 `pending_requests` 字典分發響應

### 遇到的問題和解決方案

| 問題 | 解決方案 |
|------|---------|
| 訊息監聽和響應分發衝突 | 採用「只有一個任務讀取 stdout」的設計，所有響應透過 `pending_requests` 字典分發 |
| 進程清理 | 使用 `asyncio.Event` 確保優雅停止，取消監聽任務後再終止進程 |
| 訊息解析錯誤 | 加入 JSON 解析錯誤處理，避免單一訊息失敗影響整體 |

---

## 🔧 項目 3: HTTP Server 整合

### 實作過程
- 實現 `MCPHTTPServer` 包裝器
- 整合 StreamableHTTPServerTransport
- 支援 SSE 流式響應和 JSON 響應兩種模式
- 建立路由：`/mcp/message`、`/mcp/sse`、`/mcp/session`、`/health`

### 關鍵決策
- 使用 Starlette + uvicorn 作為 HTTP 框架
- 支援有狀態和無狀態兩種 HTTP 模式
- SSE 模式作為默認，保持與 MCP 協議相容

### 測試結果
```

GET /mcp/sse - SSE 流正常建立 ✅
DELETE /mcp/session - 會話正常終止 ✅
```

## 🎯 關鍵技術決策總結

| 決策 | 理由 |
|------|------|
| 單一任務讀取 stdout | 避免讀取衝突和重複分發 |
| 延遲初始化 | 支援按需啟動，提高效能 |
| 分層架構（Registry/Discovery/Bridge） | 單一職責，易於維護和測試 |
| 雙重安全檢查 | Wrapper 層 + Bridge 層，確保安全 |
| dataclass 替代 dict | 強型別提升可維護性 |
| ThreatLevel 分級 | 支援不同程度回應 |

## 📝 特別值得記住的經驗

### 工程師蘇茉的學習

1. **Stdio 通訊的陷阱**：只有一個任務可以讀取 stdout，否則會有競爭條件
2. **Process 清理很重要**：必須先取消任務，再終止進程，否則可能產生僵屍進程
3. **安全過濾要兩層**：既要在 Wrapper 層檢查，也要在 Bridge 層檢查
4. **配置和代碼分離**：使用 ConfigManager 讓系統更靈活

### 教授蘇茉的學習

1. **模組化設計**：item5/item6 各為獨立模組，可被其他項目引用
2. **安全預設值**：嚴格 mode 預設開啟，需要明確允許才能執行
3. **文件即程式介面**：README.md 包含完整 API 文件和使用範例，降低整合門檻
4. **三層載入概念**（來自 DeerFlow 研究）：Metadata → 主體 → Resources，支援漸進式上下文載入
