## 研究背景

老爺希望工程師蘇茉研究如何優化 OpenClaw cron job 系統，參考 KerberosClaw 的 skill-cron 功能。

## 功能評估

### 1. 自然語言→cron 轉換

**功能說明**：
- skill-cron 可將「平日 9:00、18:00」自動轉換為 cron 表達式 `0 9,18 * * 1-5`

**現況分析**：
- OpenClaw CLI `openclaw cron add` 只接受標準 cron 表達式（`--cron` 參數）
- 用戶需要自己手動轉換自然語言為 cron 表達式

**可實作性**：✅ 可以實作

**實作複雜度**：⭐⭐ 中等
- 需要開發一個自然語言解析器
- 映射規則：
  - 平日/工作日 → 1-5（週一至週五）
  - 假日/週末 → 0,6（週六、週日）
  - 每天 → *
  - 早上/下午/晚上 → 對應時段

**建議實作方式**：
- 建立一個獨立的工具或整合到 CLI 中
- 輸入：`每日 9:00、18:00`
- 輸出：`0 9,18 * * *`

---

### 2. 衝突偵測確認

**功能說明**：
- 當用戶新增「平日 9:00」和「週一 9:30」時，系統會詢問如何處理衝突

**現況分析**：
- 目前 OpenClaw cron 系統沒有衝突偵測功能
- 用戶可能建立重複或時間衝突的排程

**可實作性**：✅ 可以實作

**實作複雜度**：⭐⭐⭐ 中等偏高
- 需要維護排程資料庫
- 演算法複雜度：O(n) 遍歷現有排程
- 需要 UI/對話流程來詢問用戶

**建議實作方式**：
- 在 `openclaw cron add` 前檢查時間衝突
- 發現衝突時顯示選項讓用戶選擇：
  - a. 忽略新的時間點
  - b. 取代現有時間點
  - c. 兩者都保留

---

### 3. headless-prompt

**功能說明**：
- skill-cron 在 SKILL.md frontmatter 中加入 `headless-prompt` 欄位
- 定義 skill 在排程執行時要執行的完整指令

**現況分析**：
- OpenClaw 的 cron job 已有 `payload` 機制：
  - `systemEvent`：觸發系統事件
  - `agentTurn`：執行 agent 對話
- 不需要完全複製這個概念

**可實作性**：⚠️ 部分實作 / 不需要

**說明**：
- OpenClaw 已有類似功能（payload）
- 如果要讓 skill 支援 cron 排程，可以在 skill 的 SKILL.md 中加入 `cronnable: true` 標記
- 由排程系統自動讀取 skill 的資訊並建立排程

## 結論

三個功能中有兩個（1、2）可以實作且有實用價值。第三個功能（headless-prompt）在 OpenClaw 現有架構下已有類似機制，不需要額外實作。

建議優先實作「自然語言→cron 轉換」功能，這能大幅降低用户使用 cron 的門檻。
