一開始，我只是不想再跟不同的 Agent 或是 AI 說第二次同樣的話。
每次新的對話框開始，我就要重新解釋：我用什麼工具、我踩過什麼坑、我做過什麼決定。
AI 的記性很差。這不是秘密。但我沒想到，解決記性的過程會變成一整個月的工程。
---
最早，我把所有知識塞進 prompt。
結果 context 爆了。AI 每次對話要讀幾萬 token，速度慢、成本高，而且塞越多，它越抓不到重點。
然後我把知識放進 GitHub，用純文字檔，想說至少 AI 讀得到。
但問題來了：AI 每次都要翻一百多篇筆記才能找到相關的那一篇。跟沒整理差不多。
---
四月初，Andrej Karpathy 發了一篇文章，講他怎麼建 LLM Wiki——讓模型像翻百科全書一樣查資料。
架構很好。但我實作之後發現一個他沒解決的問題：
這條知識，該放在哪一頁？
同一個 bug，有人放在「踩坑筆記」，有人放在「CDP 橋接」，有人放在「WSL2」。AI 搜尋的時候，到底該去哪找？
不是頁面不夠多。是分類方式本身有問題。
---
所以我換了一個思路：不分頁，分層。
從「AI 什麼時候需要這條知識」倒推：
第一層：你是誰。每次對話都讀。
第二層：核心事實。你住哪、用什麼工具、什麼決策。
第三層：最近在做什麼。每天自動更新。
第四層：所有的坑、方案、架構決策。搜尋時才拉。
Agent 問「你用什麼模型」讀第二層。問「CDP 橋接怎麼修」讀第四層。
不用每次翻整座百科。Token 從幾萬降到幾千。
---
分層解決了「去哪找」的問題。但最大的坑是「寫不回來」。
有一天我讓 AI 跑了一整天。CDP 橋接修了，編譯器去重做了，安全審計跑了。
做完了，我問：百科有沒有增加？
零。
修了半天水管，水龍頭從頭到尾沒開過。
從那天起，我立了一條條鐵律，寫進記憶中：做完就寫，寫完就編譯，編譯完就確認。
---
再來是搜尋的問題。
關鍵字搜尋很便宜，但「CDP」和「Chrome DevTools Protocol」是同一件事，關鍵字搜不到。
所以我加了語意搜尋。用 embedding 把知識轉成向量，搜「瀏覽器除錯」也能找到 CDP 相關的條目。
但向量搜尋有代價——每條知識 384 維度，一百多條就要存幾萬個數字。而且原文動輒幾百 token，不可能每次都塞進 context。
所以我做壓縮。每條只留標題、分類、一句話摘要、來源。AI 搜尋時先讀壓縮版，找到相關的才拉原文。
成本再降一級。
---
然後是衝突。
我的百科寫著「雲端可以同時跑 5 個模型」，同時也寫著「只能跑 3 個」。
哪個對？如果不管，AI 兩個都信。
所以我加了信任分數。每條知識 0 到 1 分。用戶定義的 1.0，驗證過的 0.9，可信的 0.7，未驗證的 0.5。互相矛盾的，AI 優先相信高分數的。
再加一個 Lint 定期掃描：哪些條目是孤兒？哪些概念被引用 5 次但沒有自己的條目？哪些互相矛盾？
跟跑測試一樣。但測的是知識庫的健康。
---
最後一個決定：從純 GitHub 轉到混合架構。
GitHub 做版本控制和備份。但 Git 不會搜尋，不會算相似度，不會自動編譯。
所以我用 Supabase 當主資料庫——pgvector 做語意搜尋，自動索引新條目，矛盾偵測跑在資料庫層。
本地還是保留純文字檔和壓縮備份。雲端掛了，本地還能跑。
AI 改壞了，git checkout 一行回滾。
---
一個禮拜下來，百科從 0 長到一百多條。每條都是做過的事、踩過的坑、驗證過的方案。
架構從「所有東西塞進 prompt」變成四層分層 + 語意搜尋 + 壓縮索引 + 信任分數 + Lint 健康檢查 + Git 回滾。
百科管的是技術知識。但我每天也在寫文章——文章也需要審查、也需要演化。所以我用同樣的機制：每篇寫完自動審查三個維度，分數存進資料庫，下一篇自動比對：分數進步了嗎？上次建議落實了嗎？
知識複利。文章也複利。
